Browser Technologies

Meer informatie

Bij het vak Browser Technologies (BT) heb ik geleerd om goed de User Experience (UX) van de gebruiker erbij te betrekken. Zo moet de website niet alleen mooi zijn, maar ook functioneel en goed bruikbaar zijn voor de gebruiker zonder problemen.

Bij dit vak heb ik veel gebruik gemaakt van formulieren. Het invullen en verwerken van formulieren bevat veel UX elementen die gebruikers moeten kunnen begrijpen. Ook is het belangrijk dat de (eventuele) fouten duidelijk zichtbaar worden weergegeven met een goede uitleg erover.

Nieuwe inzichten:

Bij dit vak heb ik geleerd hoe cruciaal een formulier moet zijn om de UX zo goed mogelijk te maken. Zo moeten de foutmeldingen goed beschreven zijn en moeten de foutmeldingen ook op een goede manier weergegeven worden, zodat de gebruiker goed kan begrijpen wat de mogelijke fout is. Ook het formulier moet goed onworpen zijn, zodat de gebruiker weet wat er gevraagd wordt en dat er geen onnodige extra vragen erbij worden weergegeven. Een voorbeeld van het slim maken van de formuliering is het toevoegen van "progressive disclosure". Dit betekent dat er vragen pas worden weergegeven als dit van toepassing is. Op deze manier zal de gebruiker geen onnodige vragen tegenkomen bij het invullen van het formulier. Ook is het belangrijk dat de website "progressive enhancement" bevat. Dit betekent dat er een backup optie moet zijn voor op het moment dat er een probleem is met een functionaliteit of de gehele website. Dit kan worden toegevoegd door een "noscript" versie van de website toe te voegen aan de website. Met "noscript" wordt er een backup versie geschreven in html, zodat de gebruiker de website, na problemen, nogsteeds kan bezoeken.

Voorheen heb ik vooral op de "happy flow" van de website gelet, waarbij er veel verschillende animaties en interacties bij komen kijken. Bij dit vak heb ik geleerd dat het ook belangrijk is om te letten op de "UX" kant van de website. Hierbij moet de gebruikerservaring zo goed mogelijk gemaakt worden. Op deze manier kunnen gebruikers de website zonder problemen de website bezoeken.

CSS control panel

Meer informatie

Ik heb bij het vak CSS veel CSS elementen gebruikt om interactie op de website mogelijk te maken. Hierbij was de challenge om geen javascript te gebruiken, maar alleen CSS. Dit was een leuke uitdaging, omdat er veel mogelijkheden zijn met CSS om interactie mogelijk te maken. Zo heb ik bijvoorbeeld gebruik gemaakt van "checkboxes" om de interactie mogelijk te maken. Ook heb ik veel gebruik gemaakt van "transitions" en "animations" om de interactie nog beter te maken voor de gebruiker. Ook heb ik geleerd om in CSS nesting te gebruiken, door te nesten houd je elementen bij elkaar en is het makkelijker om de code terug te lezen.

Nieuwe inzichten:

Ik heb bij dit vak geleerd hoe krachtig CSS kan zijn om interacties, animaties en styling te maken. Voorheen zou javascript van te pas komen om bepaalde interacties en berekeningen mogelijk te kunnen maken. Zo is het bijvoorbeeld mogelijk in css om de "checkboxes" te gebruiken om interacties tussen elementen mogelijk te maken. Hier wordt de krachtige ":has" selector voor gebruikt om de interactie mogelijk te maken. De ":has" selector kijkt op de status van de "checkbox" en veranderd zo elementen op de website gebaseerd hierop. Ook is er "calc" gebruikt om berekeningen te kunnen maken binnen CSS. Op deze manier is javascript niet meer nodig om de interactie mogelijk te maken.

Daarnaast heb ik in de code nieuwe codeer technieken geleerd en toegepast. Zo heb ik gebruk gemaakt van "CSS nesting" om de code beter te structureren en overzichtelijker te maken. Ook heb ik bij dit vak veel gebruik gemaakt van "custom properties". Deze properties zijn variabelen die gebruikt kunnen worden om dynamsich aan te passen in de CSS. Zo kan er bijvoorbeeld de kleur aangepast worden van een element als de property in combinatie met de ":has" selector wordt gebruikt.

Nieuwe inzichten:

Mijn grootste inzicht bij dit vak is het feit dat er anders gedacht moest gaan worden. De testpersoon gaf aan dat hij het fijn vond om dikke borders te hebben op de website, omdat de testpersoon een klein beetje zicht nog bevat. Deze zicht wilt de gebruiker wel nog gebruiken om informatie in zijn omgeving te kunnen ontvangen.

Het constant testen en itereren van de website heeft mij geleerd om mijn eigen ontwerpprincipes aan de kant te zetten en te focussen op de behoeftes van de testpersoon. Omdat de testpersoon, de persoon is die de website uiteindelijk gaat gebruiken.

Opbouw van een formulier

  1. Introductie tekst aan het begin
  2. Vervolgens de informatie opvragen die nodig is voor het formulier
  3. Voordat de gebruiker de antwoorden inzendt, kan de gebruiker de waarden die ingevuld zijn controleren en aanpassen
  4. Na het verzenden kan de gebruiker de fouten inzien in een samenvatting

Als de gebruiker op verzenden klikt, zal er een duidelijke bevestiging komen hiervan

  • Alle velden moeten ingevuld worden, als een veld niet verplicht is staat dit erbij
  • Er mogen geen onnodige gegevens opgevraagd worden
  • De gebruiker kan de samenvatting van de fouten bovenaan het formulier vinden. De foutmelding moet aanklikbaar zijn en ook focusbaar
  • Onder de "form submit" knop komt de tekst te staan. Hier staat dan bijvoorbeeld "vul alle velden in"
  • De gebruiker kan ook de status ervan zien, bijvoorbeeld "opslaan"
  • formulier verstuurd!

Inhoud van de formulier

  • Geef alleen de informatie die echt nodig is
  • Alle verplichte vragen moeten goed aangegeven zijn
  • Er moet goede uitleg zijn over waarom de gegevens opgevraagd worden

Design competent

  • Kleurcontrast van componenten op een pagina moet goed gecontroleerd worden. De rand moet goede contrast bevatten ten opzichte van de achtergrond
  • De componenten moeten groot genoeg zijn, zodat er geen rare overflows ontstaan
Design formulier
  • Buttons moeten aan het begin van de regel staan
  • voortgang van de gebruiker moet bovenin het formulier aangegeven worden in tekst
  • De navigatie moet consisten zijn
  • Het einde moet duidelijk aangegeven worden
  • Verplichte en niet verplichte velden moeten duidelijk aangegeven worden. Dit moet niet alleen in asterisk, maar ook in tekst
  • Buttons moeten onderaan een ul staan
Flow formulier
  • In de footer link moeten de contact voor hulp en klantenservice komen
  • Op meerdere plekken moeten er een begin en introductie getoont worden. Dit om de gebruiker duidelijk te maken wat er gevraagd wordt op het formulier
  • Voortgang moet bovenaan komen
  • De afkortingen en jargon moet uitgelegd worden in het formulier
Labels
  • De code voor label moeten altijd de zichtbare labeltekst bevatten, zodat de screenreader dit kan lezen
  • Type alles uit, gebruik geen afkortingen. Bijvoorbeeld "voornam(en)" is fout, "voornaam 1 of meerdere" is goed
  • Het is belangrijk dat de browser autocomplete hierop werkt
Beschrijving van velden
  • De belangrijke uitleg moet altijd goed zichtbaar zijn
  • Informatie moet in volgorde staan
  • Als er extra teksten bij form veld horen dan moet dit met "aria-desrcibedby" aangegeven worden
  • Ook foutmeldingen kunnen met extra tekst via "aria-describedby" gedaan worden
  • Foutmeldingen van radiogroups worden ook met "aria-describedby" aan de fieldset gekoppeld
Controleren
  • De laatste stap moet de ingevulde gegevens samenvatten en weergeven aan de gebruiker
  • Het formulier moet altijd gewijzigd kunnen worden achteraf
  • De ingevoerde gegevens mogen nooit vertaald worden

Privacy

Er wordt altijd iets verborgen zonder dat je het zelf doorhebt. Denk bijvoorbeeld aan bankgegevens.

  • Een individueel recht
    Met rust gelaten worden
  • Een sociaal recht

  • Een democratisch noodzaak
    Autonoom je mening kunnen vormen

Het wel en niet willen delen van informatie verschilt per context. Dokter? Familie? Vrienden? Werk?

Contextcollect is het samenvoegen van context. Bijvoorbeeld je hebt iemand op Facebook gezien hoe iemand is.

Privacy verschuift steeds. Vroeger deelde je best weinig, omdat er minder te delen viel. Nu wordt er meer gedeelt, bijvoorbeeld met de overheid.

“Never reveal your identity”

Ook het IP adres wordt in de wet onder persoonsgegevens gezien. Als de gegevens niet gekoppeld kan word aan de persoon dan zijn het geen persoonsgegevens. Als dit wel zo is dan wordt het AVG ingesteld.

Je moet een geldige reden hebben om gegevens te mogen.
Toestemming vragen voor de gegevens is belangrijk. En het moet ook geïnformeerd zijn. Bijvoorbeeld laten zien met wie het wordt gedeeld en waarvoor. De gegevens moeten ook altijd ingetrokken kunnen worden.

Als gegevens noodzakelijk zijn voor het uitvoeren van een overeenkomst

  • Wettelijk verplicht, dus overheid
  • Bij vitaal belang is dat ook belangrijk. Bijvoorbeeld als iemand in gevaar is, dan kun je de telefoon gebruiken
  • Bij algemeen belang of voor het openbaar gezag. Bijvoorbeeld politie
  • Bij een gerechtvaardigd belang. Bijvoorbeeld nodig om dienst uit te kunnen voeren. Het beste is om mensen om toestemming te vragen

Doelbinding. als je data hebt verzameld dan mag dit niet zomaar gedeeld worden. Dus alleen geldig voor doel 1 en niet doel 2.

Rechten

  • Je mag altijd je gegevens inzien en corrigeren en aanvullen en verwijderen
  • Je hebt recht op vergetelheid. Bijvoorbeeld oude data
  • Data mag meegenomen worden van de ene partij naar de andere partij. Bijvoorbeeld als je Spotify abonnement hebt, dan moet je dat mee kunnen nemen naar een andere dienst. Het moet op een digitale manier, makkelijk overgezet kunnen worden
  • Door de AVG is data over personen niet alleen maar een asset…
  • Data over personen is ook een liability.
  • Goed nadenken over de gegevens die je verzameld. Hier zijn bedrijven aansprakelijk voor.
  • Dataminimalisatie is belangrijk om ervoor te zorgen dat er niet teveel data wordt verzameld.

Achttal strategieën om privacy te waarborgen:

  1. Minimise
  2. Separate
  3. Inform
  4. Control
  5. Abstract
  6. Demonstrate
  7. Hide
  8. Enforce

The Web You Want

The Web You Want (deel 1)

"The Web You Want" richtte zich op de meningen en argumenten van verschillende experts in het werkveld. De experts willen vooral dat het web toegankelijk en inclusief wordt voor iedereen.

Tijdens de conferentie werd duidelijk dat het web een belangrijk onderdeel is van ons leven en dat iedereen toegang moet hebben tot het web.

Een spreker die mij bij is gebleven is Niels Leenheer. Hij liet zien dat er heel veel dingen mogelijk zijn met het web. Zelfs dingen die je niet zou verwachten. Zo liet let Niels zien dat er met CSS een klok gemaakt kan worden in een "osciloscoop". Dit apparaat wordt normaal gesproken gebruikt voor het meten van electrische signalen. Niels liet ons ook zien dat hij een klok heeft kunnen maken met lasers. Dit is een hele mooie toepassing van het web.

De gastsprekers willen vooral dat het web accessible wordt en ook inclusief voor veel mensen. Ook worden er veel argumenten gegeven waarom het web toegankelijk en inclusief moet zijn. Zo is het web een belangrijk onderdeel van ons leven en is het belangrijk dat iedereen toegang heeft tot het web, ongeacht hun beperkingen of omstandigheden. Ook wordt er benadrukt dat het web een plek moet zijn waar iedereen zich welkom voelt en waar iedereen kan deelnemen aan de online gemeenschap.

The Web You Want (deel 2)

Tijdens het tweede deel van de conferentie van "The Web You Want" werd er nog dieper ingegaan op de web accesibility en inclusiviteit. Voor wie bouwen wij het web eigenlijk, en welke middelen worden er gebruikt? Het was interessant om te zien dat Artificial Intelligence (AI) een positieve bijdrage kan leveren aan het web.

Léonie Watson heeft gesproken over de rol van AI in het web en er buiten. Léonie is zelf blind, waardoor zij vaak hulpmiddelen gebruikt om haar omgeving te kunnen begrijpen. Tegenwoordig wordt AI steeds vaker gebruikt om te helpen bij taken, zo gebruikt Léonie AI om haar omgeving beter te kunnen begrijpen. Met "Meta AI glass" kan Léonie haar omgeving laten beschrijven en ook objecten vinden. In dit geval vervangt AI de taken van de mens, doordat AI nu de website beschrijft. De accesibility kant hiervan is het gebruik van "alt labels". Dit wordt niet altijd gedaan, waardoor AI de taak ervan over kan nemen, zodat er alsnog een beschrijving naar voren kan komen.
AI heeft ook negatieve impact op het web. Zo worden gebruikers vaak lui, omdat er minder werk verricht hoeft te worden door AI. AI zorgt er ook voor dat de gebruiker de AI steeds meer begint te vertrouwen op wat het zegt. Als AI direct wordt gebruikt, zonder er zelf over na te denken dan kan het voorkomen dat de gebruiker niks weet van wat de gebruiker gedaan heeft om een bepaalde doel te bereiken met AI.

Meesterproef

{Ink}

De uitkomst van het project

Als afsluiting van de minor hebben we met ons team gewerkt aan het project voor {Ink}. Ink heeft ons gevraagd om de huidige training van Ink te verbeteren. De training gaat erover hoe gebruikers kunnen interacteren met mensen met een beperking. Het probleem dat ervaren werd, was het feit dat mensen erg snel afhaakten. Dit probleem hebben wij als team proberen op te lossen door minigames te maken voor de verschillende trainingen.

Mijn inbreng binnen het project

Tijdens dit project heb ik een grote rol gespeeld in de performance en de animaties van de webpagina's. Mijn belangrijkste bijdragen waren:

  • Performance & PWA: Omdat onze applicatie veel zware afbeeldingen bevat, heb ik Progressive Web App (PWA) technieken toegepast. Door de website lokaal te cachen, hoeven zware pagina's niet bij elke klik opnieuw in te laden, wat de laadtijd en gebruikerservaring enorm heeft verbeterd.
  • User Testing: Ik heb het initiatief genomen om ons prototype te testen in de praktijk (onder andere door een user test met mijn oom). De data uit deze test heb ik vertaald naar concrete verbeterpunten voor het team en succesvol uitgelegd bij de opdrachtgever.
  • Development: Daarnaast heb ik mij bezig gehouden met het maken van animaties en transities op de pagina's en ook tussen pagina's. Ook de gehele infrastructure heb ik in elkaar gezet, waarbij de pagina's ook met elkaar gelinkt zijn en componenten dynamisch veranderen gebaseerd op de webpagina van de gebruiker.

Nieuwe inzichten

De Meesterproef heeft me ontzettend veel nieuwe inzichten gegeven, vooral op het gebied van samenwerken in een team. Ik heb geleerd dat strakke communicatie en het maken van gezamenlijke codeer-regels handig zijn om een repository schoon te houden. Technisch gezien heeft de implementatie van de PWA me echt laten inzien dat performance van de website ook ertoe doet.

conclusie & reflectie

Reflectie

Grootste leermomenten

Tijdens de minor heb ik geleerd hoe belangrijk het is om een website robuust te maken. Dit heb ik bijvoorbeeld gedaan door de code zo te schrijven dat er amper javascript in voor hoeft te komen. Hierdoor kan de website blijven functioneren zonder javascript. Ook de toegankelijkheid van de website is erg belangrijk, omdat op deze manier de inclusiviteit van de website gewaarborgd blijft. Zo kan iedereen de website gebruiken, ongeacht de beperking.

Lastige momenten

Tijdens de minor heb ik ook lastige momenten ervaren. Zo was het soms lastig om ook aan de toegankelijkheid van de website te denken. Ook moest ik verschillende keren de performance van de website goed monitoren, zodat de website goed bleef functioneren. Ook is de performance belangrijk om de User experience hoog te houden, omdat de gebruiker dan niet hoeft te wachten op het laden van een pagina.

Troste momenten

Als ik terug kijk op de minor, dan heb ik veel nieuwe dingen geleerd. Zo ben ik anders gaan kijken naar de manier waarop javascript gebruikt wordt. Ook de toegankelijkheid in het proces van het maken van een website is erg belangrijk. Zo ben ik trots op het maken van robuuste code, zodat de website zonder javascript ook kan blijven functioneren.

Weekly nerd

De verschillende inzichten van de weekly nerds hebben mij een beter beeld gegeven van de verschillende aspecten van het web. De interessante aspecten van de weekly nerds waren vooral de toegankelijkheid en de privacy op het web. Zo was er 1 spreker die een server op heeft gesteld, zodat er onafhankelijk van andere partijen een website gehost kan worden. Hierdoor wil ik zelf ook meer privacy en toegankelijk gericht ontwerpen, zodat de gebruikservaring van de gebruiker verbeterd. Ook zal de privacy van de gebruiker goed gewaarborgd kunnen blijven.

What is next?

In de toekomst wil ik vooral verder gaan met het maken van robuuste code. Hierbij moet de code robuust zijn voor de gebruiker, maar ook voor andere developers. De code moet naast robuust ook goed toegankelijk zijn, zodat de websites inclusief kunnen zijn voor iedereen.
